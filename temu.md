## Temu 核价插件执行策略

- 采集价/原价，可根据页面实时捕获计算/API对接自己的产品库/对接商品渠道来源，辅助作为数据来源

一句话定位：本文用于沉淀 Temu 核价插件的申报价、复核底线、倍率阶梯、人工介入和异常兜底规则，支撑批量上品前的价格判断。

> **合规边界** ：本文中的“核价策略”只能用于保证申报价、商品事实、图片、SKU、成本资料和履约能力之间的一致性，不能用于伪造 BOM / 发票 / 成本资料、故意错放类目、隐瞒商品属性、规避平台禁限售或误导买手。所有图片、标题、属性和资料调整都必须保留真实商品事实；无法证明的内容进入人工复核或放弃。

### 一、文档目标与适用范围

本文档只保留 Temu 核价插件执行策略，以及插件执行必须用到的规则口径。

重点解决：（26美金的临界值为模拟计算，可根据实际情况进行调整）

- 26 美金以下商品如何计算首次申报价和复核底线。
- 26 美金以上商品如何按店铺 / 店铺组配置倍率阶梯。
- 混合价格段 SKU 如何按价格段分别执行低价品和高价品策略。
- 插件如何判断通过、继续复核、终止、转人工或换图重试。

本文档不展开核价插件执行之外的运营背景。

### 二、平台认知与场景

### 1\. 核价系统的三层架构

Temu 的核价不是单一决策，而是三层叠加。插件的所有策略，本质上都围绕这三层的 [信息不对称](https://zhida.zhihu.com/search?content_id=278425163&content_type=Article&match_order=1&q=%E4%BF%A1%E6%81%AF%E4%B8%8D%E5%AF%B9%E7%A7%B0&zhida_source=entity) 展开。

| 层级 | 主导方 | 核心逻辑 | 插件可利用的空间 |
| --- | --- | --- | --- |
| 系统比价层 | AI 算法 | 图像识别 + [文本匹配](https://zhida.zhihu.com/search?content_id=278425163&content_type=Article&match_order=1&q=%E6%96%87%E6%9C%AC%E5%8C%B9%E9%85%8D&zhida_source=entity) ，抓取全网同款/近似款价格作为锚点 | [视觉指纹](https://zhida.zhihu.com/search?content_id=278425163&content_type=Article&match_order=1&q=%E8%A7%86%E8%A7%89%E6%8C%87%E7%BA%B9&zhida_source=entity) 规避、类目错位、属性微调、关键词替换 |
| 买手人工层 | 类目买手 | 在系统建议价基础上，结合 KPI、库存压力、活动规划做人工调整 | 报价节奏、 [成本资料弹性](https://zhida.zhihu.com/search?content_id=278425163&content_type=Article&match_order=1&q=%E6%88%90%E6%9C%AC%E8%B5%84%E6%96%99%E5%BC%B9%E6%80%A7&zhida_source=entity) 、商务关系、时间窗口 |
| 成本验证层 | 风控/质检 | 异常价格或品类要求提供 BOM、报价单、物流发票等佐证 | BOM 拆分策略、成本模板生成 |

**核心原则** ：系统负责找锚点，买手负责定价权，成本资料负责兜底验证。三层中任意一层出现”信息不对称”，就有博弈空间。

### 2\. 核价的底层逻辑：同款比价

Temu 核价的核心算法是 **同款比价** 。买手手上通常有同款或近似款的行业底价，如果产品一眼就能被系统找到对标，核价基本贴着成本线甚至更低。

**定价锚点的可控性** ，决定了核价空间的大小：

| 产品类型 | 锚点可控性 | 核价空间 | 插件策略重心 |
| --- | --- | --- | --- |
| 公模标品（手机壳、数据线、基础家居） | 极低 | 几乎无 | 脱锚为主（换图、改属性、错类目） |
| 微差异化产品（改尺寸、加配件、换包装） | 中 | 有限 | 视觉指纹 + 属性微调 |
| 私模/外观专利款 | 高 | 大 | 正常申报，减少不必要的博弈 |
| 认证门槛品（电器、儿童用品、美妆工具） | 中高 | 较大 | 认证即壁垒，常规申报即可 |
| 非标设计品（创意家居、个性化配饰） | 高 | 大 | 强调设计属性，避开标品池 |
| 时效性/季节性新品 | 视窗口而定 | 窗口期大 | 抢时间窗口，快速占位 |

### 3\. 核价场景分类

插件需识别不同场景，匹配不同策略强度：

**场景 A：公模红海标品**

- 特征：1688/淘宝一搜一大把，主图高度雷同。
- 系统行为：比价锚点极多，首次申报即压到底价。
- 插件策略：必须启用脱锚策略（换图、属性微调、类目错位），否则无利润空间。

**场景 B：微差异化产品**

- 特征：公模基础上改了尺寸、加了配件、换了包装。
- 系统行为：可能识别为近似款，也可能因规格差异无法直接匹配。
- 插件策略：先正常申报，若被拒价再启动脱锚策略。

**场景 C：认证门槛品**

- 特征：需要 CE/FCC/CPC 等认证，供应链门槛较高。
- 系统行为：比价池相对小，买手对合规供应商容忍度高。
- 插件策略：强调认证属性，正常申报，不主动降倍率。

**场景 D：时效性/季节性新品**

- 特征：平台新开类目或主推新品类，市场供给尚未饱和。
- 系统行为：比价数据不足，核价容忍度高。
- 插件策略：抢时间窗口，首次申报可适度激进。

**场景 E：大促前锁价期**

- 特征：Black Friday、Prime Day 等前 2-4 周，平台需要备货。
- 系统行为：买手背 GMV 和库存 KPI，低价通过率提升。
- 插件策略：accept\_ratio 临时下调 0.1-0.2，加速通过。

### 4\. 平台风控趋势与生命周期

插件策略不是静态的，必须考虑平台风控的进化方向：

| 风控维度 | 当前状态 | 进化趋势 | 对插件的影响 |
| --- | --- | --- | --- |
| 图像回溯 | 已有，但覆盖不全 | 会越来越强 | 换图策略的窗口期在缩短 |
| 规格关联 | 依赖文本匹配 | 会向多维参数匹配发展 | 属性微调的阈值会越来越严 |
| 店铺关联 | 已有法人/设备/IP 风控 | 会向供应链关联发展 | 多主体矩阵的风险在上升 |
| 成本核验 | 部分类目抽检 | 会向大数据交叉验证发展 | BOM 拆分需在更合理范围内 |
| 买手轮岗 | 频繁 | 会持续 | 单一买手关系不可持续 |

**结论** ：信息博弈（换图、改属性）的 ROI 在递减，结构性优势（私模、认证、独家供应链）的 ROI 在递增。插件的终极价值不是”骗过系统”，而是把已验证有效的博弈手段自动化、规模化，同时沉淀数据反向指导产品策略。

### 三、核价手段全景

### 1\. 信息博弈（对抗系统比价层）

系统比价层的本质是”图像识别 + 关键词/规格匹配”，它的盲区就是信息博弈的空间。

**手段 1：主图视觉指纹规避**

| 具体做法 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- |
| 换拍摄角度（正面 → 45 度斜角 → 俯视 → 场景使用图） | 降低 AI 识图匹配度 | 高 | 低 |
| 换背景与场景（白底 → 家居场景 → 户外场景 → 纯平面设计图） | 改变图片特征向量 | 高 | 低 |
| 加配件展示（单品 → 单品+收纳袋/电池/贴纸） | 让系统识别为”套装” | 高 | 低 |
| 换包装视觉（裸品 → 品牌包装 → 礼盒装） | 增加视觉差异点 | 高 | 低 |
| 加手模/真人使用图（无模特 → 手部持握 → 真人使用） | 改变画面主体 | 中 | 低 |
| 主图加遮挡/装饰元素 | 干扰边缘识别 | 中 | 中（可能触发”图片与产品不符”） |
| 图文排版变化（纯产品 → 产品+卖点标签 → 产品+尺寸标注） | 改变图像区域分布 | 低 | 低 |

**手段 2： [SKU 规格参数微调](https://zhida.zhihu.com/search?content_id=278425163&content_type=Article&match_order=1&q=SKU+%E8%A7%84%E6%A0%BC%E5%8F%82%E6%95%B0%E5%BE%AE%E8%B0%83&zhida_source=entity)**

| 具体做法 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- |
| 改计量单位（”个” → “组/套装”，”单件” → “2 件装”） | 比价模型无法直接匹配 | 高 | 低 |
| 微调试尺寸/重量（±1-2 cm，±10-20 g） | 跳出标准规格库 | 高 | 低 |
| 颜色/款式微调（增加”限量色”或”组合色”） | 识别为新的 SKU 变体 | 中 | 低 |
| 材质描述升级（”塑料” → “ABS 工程塑料”） | 关键词匹配池变化 | 中 | 低 |
| 加不可比 SKU（多规格链接里放一个”豪华版”） | 拉高整体价格带 | 中 | 中（需实际可交付） |

**手段 3： [类目与关键词错位](https://zhida.zhihu.com/search?content_id=278425163&content_type=Article&match_order=1&q=%E7%B1%BB%E7%9B%AE%E4%B8%8E%E5%85%B3%E9%94%AE%E8%AF%8D%E9%94%99%E4%BD%8D&zhida_source=entity)**

| 具体做法 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- |
| 切换细分类目（露营椅 → 户外便携折叠椅） | 切换比价池 | 中 | 中（可能触发”类目错放”） |
| 标题关键词替换（camping chair → outdoor folding chair） | 改变文本匹配锚点 | 中 | 低 |
| 场景化关键词（phone case → protective cover / fashion accessory case） | 跳出 3C 标品池 | 中 | 低 |

### 2\. 商务博弈（对抗买手人工层）

买手不是机器，他们背 KPI（动销率、GMV、利润指标），存在人性的博弈空间。插件不介入灰色操作，但可以把”阳谋”层面的商务博弈结构化、自动化。

**手段 1：报价节奏控制**

| 具体做法 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- |
| 首次报价留足 30%-50% 压缩空间 | 避免被买手一刀砍死 | 高（阶梯倍率） | 低 |
| 分批次让步，每次让步要求流量/档期承诺 | 以退为进，换取资源 | 中（人工谈判，插件记录） | 低 |
| 制造稀缺性（产能有限、原材料上涨、其他平台也在谈） | 给买手心理压力 | 低（纯人工） | 低 |
| 利用竞品售价做锚（”亚马逊卖 15，你核 3 我怎么供货”） | 抬高买手的心理底价 | 中 | 低 |

**手段 2：成本资料弹性**

| 具体做法 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- |
| BOM 表拆分（原材料、加工费、包材费分项列示） | 让成本看起来更高 | 中（模板） | 中（需在合理范围） |
| 加入 3%-5% 损耗系数 | 合理放大原材料成本 | 中 | 低 |
| 加工费分拆为多道工序（注塑、组装、质检、包装） | 工序越细，总价越容易做高 | 中 | 低 |
| 包材升级（彩盒替代白盒，逐项列示内盒、外箱、标签、说明书） | 包材费用大幅增加 | 中 | 低 |
| 头程物流按实际发票或预估高值计入 | 增加硬成本项 | 中 | 低 |
| 加入质检预留（2%-5%）和管理摊销 | 摊入更多固定成本 | 中 | 低 |
| 多工厂报价策略（配合度高的工厂开”高报价单”用于审核） | 提供平台可接受的”佐证” | 低（纯人工） | 中（涉及商业诚信边界） |

**手段 3：链接养起来后的提价博弈**

| 具体做法 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- |
| 新品期微利/平本做数据，养动销和评分 | 建立链接权重 | 低（纯人工） | 低 |
| 动销稳定后以”成本上涨”“工厂调价”为由申请提价 | 利用链接权重换谈判筹码 | 低（人工操作） | 中（可能被拒或要求证明） |
| 短暂断供后重新上架，要求重新核价 | 重置失败倍率 | 低（人工操作，插件记录） | 中（平台可能降权） |

### 3\. 组织套利（结构性空间）

利用平台各单元之间的信息不完全互通，制造套利空间。

| 具体做法 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- |
| 多主体矩阵申报（不同法人、不同营业执照、不同 IP） | 测试不同买手的核价口径 | 中 | 高（关联风控） |
| 多店铺并行申报（同时向多家提报，取最优结果） | 最大化首次通过率 | 中 | 高（关联风控） |
| 顺序申报（一家失败后冷却，换另一家） | 分散风险，避免批量关联 | 中 | 中 |
| 跨站点申报（US、EU、JP 核价团队不互通） | 利用站点间核价差异 | 中 | 中 |
| 跨平台套利（用亚马逊售价作为市场参考价谈判） | 拉高 Temu 买手的心理锚点 | 低（人工谈判） | 低 |

### 4\. 时间博弈（节奏控制）

时间窗口会显著影响系统容忍度和买手决策空间。

| 具体做法 | 时间特征 | 目标 | 插件化程度 | 风险 |
| --- | --- | --- | --- | --- |
| 新品窗口期占位 | 新开类目或主推新品类的前 30-60 天 | 系统比价数据不足，核价容忍度高 | 高 | 低 |
| 活动前锁价 | 大促前 2-4 周（Black Friday、Prime Day 等） | 平台需要备货，accept\_ratio 可临时下调 | 高 | 低 |
| 季度末冲量 | 每季度最后 10-15 天 | 买手背 GMV KPI，低价通过率提升 | 高 | 低 |
| 淡季暂停 | 1-2 月、7-8 月 | 避免平台压价期硬碰 | 高 | 低 |
| 断供博弈 | 链接养起来后的任意时点 | 以”停产/成本暴涨”为由短暂下架，重新上架时重置核价 | 低（人工操作，插件记录） | 中（平台可能降权） |
| 买手轮岗后重新提报 | 买手变动后 1-2 周 | 新买手核价标准和宽松度可能不同 | 低（人工判断） | 低 |

### 5\. 数据博弈（链接权重反向施压）

这一层不是对抗系统或买手，而是利用平台的运营规则反向获取定价权。

### 四、插件执行策略

### 4.1 执行总流程与变量口径

**执行总流程**

| 步骤 | 动作 | 关键判断 | 输出 |
| --- | --- | --- | --- |
| Step 1 | 读取商品 / SKU 信息 | 是否能拿到 SKU、采集价 X、店铺 / 店铺组 | 可核价 SKU 清单 |
| Step 2 | 判断价格段 | X ≤ 26 或 X > 26 | 进入低价品策略或高价品策略 |
| Step 3 | 读取店铺配置 | 店铺 / 店铺组是否有倍率配置 | quote\_steps、accept\_ratio、min\_ratio、max\_rounds |
| Step 4 | 生成申报价 | 按价格段和配置生成 P0 或当前轮申报价 P | 当前申报价 |
| Step 5 | 获取系统返回值 | 系统返回值 R 是否达到接受条件 | 通过 / 继续复核 / 终止 |
| Step 6 | 处理异常 | 价格、SKU、返回值是否异常 | 人工复核 / 放弃 |
| Step 7 | 记录结果 | 保存价格、倍率、返回值、动作和原因 | 核价记录 |

**变量口径**

| 变量 | 含义 |
| --- | --- |
| X | 采集价 / 采购价 |
| k | 26 美金以下商品首次申报使用的最小倍率 |
| P0 | 首次申报价 |
| P | 当前轮申报价 |
| Pn | 第 n 次复核申报价 |
| Pmin | 可接受底线价 |
| R | 系统返回值 / 平台返回价 |
| quote\_steps | 26 美金以上商品使用的阶梯倍率列表 |
| accept\_ratio | 26 美金以上商品的接受底线倍率 |
| min\_ratio | 26 美金以上商品的终止倍率 |
| max\_rounds | 最大复核次数 |
| fail\_action | 倍率走完后的失败动作 |

### 4.2 低价品执行策略（X ≤ 26）

**适用范围** ：采集价 **X ≤ 26** 的商品。

**首次申报规则**

26 美金以下商品首次申报采用”40 美金价格带”策略。

```
P0 = X × k
P0 > 40.1
```
- X 为采集价 / 采购价。
- k 为满足 `X × k > 40.1` 的最小倍率。
- k 按 `0.1` 为最小步进向上取值。
- 插件计算时先用 `40.1 / X` 得到理论倍率，再向上取到最近的 `0.1` 倍率档；若仍不满足，继续增加 `0.1` 直到满足首次申报条件。

**倍率示例**

| 原价 X | 最小倍率 k | 首次申报价 P0 |
| --- | --- | --- |
| 1 | 40.2 | 40.2 |
| 2 | 20.1 | 40.2 |
| 3 | 13.4 | 40.2 |
| 5 | 8.1 | 40.5 |
| 10 | 4.1 | 41.0 |
| 15 | 2.7 | 40.5 |
| 20 | 2.1 | 42.0 |
| 26 | 1.6 | 41.6 |

**复核底线与接受条件**

低价品首次申报看 40 美金价格带，复核接受看利润底线。

```
Pmin = X + 6
Pn = P0 × 0.9^n
```
- Pmin 为可接受底线价。
- Pn 为第 n 次复核申报价，n 从 1 开始计算（不包含首次申报价 P0）。

| 判断项 | 规则 | 处理 |
| --- | --- | --- |
| 接受条件 | R ≥ X + 6 | 接受当前返回价 |
| 继续复核 | R < X + 6，且 Pn ≥ X + 6 | 继续下一轮复核 |
| 终止条件 | Pn < X + 6 | 停止复核，进入人工复核 / 放弃 |

注意：本策略不使用”底线价保证单件 26 美金”的口径。如需 26 美金保底，应单独调整为 `Pmin = max(26, X + 6)` 并重新确认低价品策略。

### 4.3 高价品阶梯执行策略（X > 26）

**适用范围** ：采集价 **X > 26** 的商品。

**核心原则**

26 美金以上商品不写死某一套倍率。插件只执行核价流程，具体倍率由店铺或店铺组配置。

- 每个店铺 / 店铺组可配置自己的阶梯倍率和接受底线倍率。
- 每轮申报价 = **X × 当前倍率** 。
- 当 **R ≥ X × accept\_ratio** 时，当前 SKU 通过。
- 当 **R < X × accept\_ratio** 时，进入下一档倍率。
- 倍率走完后仍未通过，进入人工复核或放弃。

**配置字段**

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| 店铺 / 店铺组 | 店铺 A / 店铺组 B | 策略绑定对象，不同店铺可使用不同阶梯 |
| 价格段 | X > 26 | 当前规则只适用于 26 美金以上商品 |
| quote\_steps | 12,10,8,6,4,3,2,1.6,1.5,1.4,1.3,1.2,1.1 | 阶梯倍率列表，示例不代表固定规则 |
| accept\_ratio | 1.3 | R 达到该倍率对应价格即通过 |
| min\_ratio | 1.1 | 低于该倍率不再继续申报 |
| max\_rounds | 13 | 最大复核次数 |
| fail\_action | 人工复核 / 放弃 | 倍率走完后的动作 |

**执行逻辑**

```
输入：采购价 X、SKU 信息、店铺 / 店铺组、quote_steps、accept_ratio、min_ratio、max_rounds

1. 判断 X > 26，进入高价品阶梯核价流程
2. 按 quote_steps 从高到低逐轮申报
3. 当前申报价 P = X × quote_steps[n]
4. 获取系统返回值 R
5. 如果 R ≥ X × accept_ratio，则当前 SKU 通过
6. 如果 R < X × accept_ratio，且还未到最后一轮，进入下一倍率
7. 如果倍率走完、当前倍率低于 min_ratio，或达到 max_rounds，进入人工复核 / 放弃
```

**配置示例**

| 场景 | quote\_steps | accept\_ratio | min\_ratio | 说明 |
| --- | --- | --- | --- | --- |
| 示例 A | 12,10,8,6,4,3,2,1.6,1.5,1.4,1.3,1.2,1.1 | 1.3 | 1.1 | 倍率可从较高档开始，不代表必须从 12 倍起步 |
| 示例 B | 4,3,2,1.6,1.5,1.4,1.3 | 1.3 | 1.3 | 只需要中等倍率阶梯时可参考 |
| 示例 C | 3,2.5,2,1.8,1.5 | 1.5 | 1.5 | 接受底线更高的店铺 / 店铺组可参考 |
| 示例 D | 1.5,1.4,1.3,1.2 | 1.3 | 1.2 | 低倍率阶梯可参考 |

**示例流程**

假设：X = 30，店铺配置 quote\_steps = `12,10,8,6,4,3,2,1.6,1.5,1.4,1.3` ，accept\_ratio = `1.3` ，底线价格 = `39` 。

| 轮次 | 当前倍率 | 申报价 P | 判断条件 | 处理 |
| --- | --- | --- | --- | --- |
| 1 | 12 | 360 | R ≥ 39 | 通过，否则进入 10 倍 |
| 2 | 10 | 300 | R ≥ 39 | 通过，否则进入 8 倍 |
| … | … | … | … | … |
| 8 | 1.6 | 48 | R ≥ 39 | 通过，否则进入 1.5 倍 |
| 9 | 1.5 | 45 | R ≥ 39 | 通过，否则进入 1.4 倍 |
| 10 | 1.4 | 42 | R ≥ 39 | 通过，否则进入 1.3 倍 |
| 11 | 1.3 | 39 | R ≥ 39 | 通过，否则人工复核 / 放弃 |

### 4.4 混合价格段 SKU

同一链接下如果同时存在 26 美金以下和 26 美金以上 SKU：

- 小于等于 26 美金的 SKU 走低价品执行策略。
- 大于 26 美金的 SKU 走高价品阶梯执行策略。
- 每个 SKU 单独保留申报价、系统返回值、处理结果和异常原因。

### 4.5 返回值驱动的脱锚动作分支

系统返回值 `R` 不是单一数字，背后携带平台的核价判定原因。插件在拿到 `R` 后，先解析返回原因，再决定下一动动作。

**返回值分级**

| 级别 | 返回值特征 | 含义 | 插件动作 |
| --- | --- | --- | --- |
| 通过 | R 达到当前策略接受条件 | 核价成功 | 接受，记录结果，流程结束 |
| 压价 | R 低于接受条件，但高于终止条件 | 系统有意愿，但要求降价 | 进入下一档复核价或换图重试 |
| 拒价 | R 极低或无有效价格返回 | 系统识别到强锚点，直接拒绝 | 触发换图/改属性/转人工 |
| 异常 | 返回超时、格式错误、SKU 状态变更 | 技术或状态异常 | 记录异常，稍后重试或转人工 |

**压价场景下的动作选择**

```
1. 判断当前是否已到达 max_rounds 或 min_ratio
   - 是 → 进入人工复核 / 放弃
   - 否 → 继续下一步

2. 判断当前轮次是否支持换图重试
   - 若店铺配置 allow_retry = true，且当前 SKU 有备用图包
   - 则执行换图重试
   - 否则进入下一倍率申报

3. 判断当前轮次是否支持 SKU 属性微调
   - 若店铺配置 allow_sku_adjust = true，且当前类目支持规格微调
   - 则执行属性微调后重新提报
   - 否则进入下一倍率申报
```

**拒价场景下的动作选择**

拒价通常意味着系统已识别到强同款锚点。此时插件不继续降倍率，直接触发”脱锚策略”：

```
1. 检查是否有备用图包 / 备用属性方案
   - 有 → 执行换图重试或属性微调后重新提报
   - 无 → 进入人工复核

2. 人工复核时，运营人员可选择：
   - 补充差异化素材后重新上架
   - 放弃当前 SKU
   - 转移至其他店铺 / 站点重新提报（见 4.6.3 多主体矩阵）
```

**视觉指纹规避与换图重试**

系统比价层依赖图像识别和文本匹配来找锚点。插件通过控制”视觉指纹”降低被系统关联到同款低价的概率。

触发条件：店铺配置 `allow_retry = true` ；当前 SKU 有备用图包；当前处于压价或拒价状态；同一 SKU 换图次数未达上限（默认 `max_image_retry = 3` ）。

备用图包按以下维度做差异化（优先级从高到低）：

| 维度 | 策略 |
| --- | --- |
| 主图角度 | 正面图 → 45 度斜角图 → 俯视图 → 场景使用图 |
| 背景与场景 | 白底图 → 家居场景图 → 户外场景图 → 纯平面设计图 |
| 配件展示 | 单品图 → 单品+收纳袋 → 单品+电池/贴纸/小配件 |
| 包装视觉 | 裸品图 → 带品牌包装图 → 礼盒装图 |
| 模特/手模 | 无模特 → 手部持握图 → 真人使用图 |
| 图文排版 | 纯产品图 → 产品+卖点标签图 → 产品+尺寸标注图 |

执行流程：检查换图次数 < max\_image\_retry → 取出下一优先级的备用图替换主图 → 保持采购价 X 和申报价 P 不变重新提报 → 获取新的系统返回值 R’ → 达到接受条件则通过并记录”换图成功”；否则 image\_retry\_count + 1，继续换图或进入下一倍率。

换图边界：不改变采购价 X 和申报价 P；若触发”图片与产品不符”或质检异常，立即终止转人工；同一 SKU 换图记录需保留，用于后续分析。

**SKU 属性微调与重新提报**

当换图无法脱锚时，插件可尝试在规格参数层面制造”不可比性”。

可用维度（优先级从高到低）：

| 维度 | 微调方式 | 系统影响 |
| --- | --- | --- |
| 计量单位 | “个” → “组/套装”，”单件” → “2 件装” | 比价模型可能无法直接匹配 |
| 尺寸/重量 | 长度 ±1-2 cm，重量 ±10-20 g | 跳出标准规格库锚点 |
| 颜色/款式 | 增加一种”限量色”或”组合色” | 系统识别为新的 SKU 变体 |
| 配件差异 | 基础款 → 基础款+收纳袋 | 识别为套装而非单品 |
| 材质描述 | “塑料” → “ABS 工程塑料” | 关键词匹配池变化 |

触发条件：店铺配置 `allow_sku_adjust = true` ；当前类目允许规格微调；当前处于拒价状态或换图已耗尽；同一 SKU 属性调整次数未达上限（默认 `max_sku_adjust = 2` ）。

执行流程：检查调整次数 < max\_sku\_adjust → 按优先级选择可调维度生成新 SKU 属性组合 → 重新计算申报价 P = X × 当前倍率（倍率不因属性微调而重置）→ 重新提报 → 获取新的系统返回值 R’ → 达到接受条件则通过并记录”属性微调成功”；否则 sku\_adjust\_count + 1，继续下一维度或进入下一倍率。

微调边界：微调后的产品必须实际可交付，不能出现”图片与实物不符”的客诉风险；微调后的 SKU 需同步更新库存和供应链信息；若平台要求提供”新旧款关系证明”，转人工处理。

**类目与关键词错位配置**

类目标签和标题关键词决定了系统去哪个”比价池”找锚点。插件支持在提报时选择差异化类目和关键词策略。

| 原始类目 | 错位目标 | 适用场景 |
| --- | --- | --- |
| 露营椅 | 户外便携折叠椅 | 锚点池从红海露营赛道切换 |
| 手机壳 | 创意保护套 / 个性化配件 | 降低标品比价密度 |
| 基础家居 | 创意家居 / 收纳整理 | 切入设计驱动型比价池 |

| 原始关键词 | 替换关键词 | 效果 |
| --- | --- | --- |
| camping chair | outdoor folding chair / portable lounger | 锚点关联度下降 |
| phone case | protective cover / fashion accessory case | 跳出 3C 标品池 |
| storage box | organizer bin / closet container | 场景化关键词，弱化标品属性 |

执行口径：类目错位和关键词替换需在店铺配置中预置 `category_alternatives` 和 `keyword_alternatives` ；插件不自动猜测类目，只执行配置表中已有的映射；首次提报使用原始类目，换图/属性微调后仍未通过时才启用错位配置；错位后的 SKU 需人工确认是否符合平台类目规范，避免被判定为”类目错放”而降权。

### 4.6 高级配置

### 4.6.1 时间窗口与申报节奏控制

核价不是纯数学问题，时间窗口会显著影响系统容忍度和买手决策空间。

| 窗口类型 | 时间特征 | 策略 |
| --- | --- | --- |
| 新品窗口期 | 平台新开类目或主推新品类的前 30-60 天 | 核价容忍度高，首次申报可适度激进 |
| 活动前锁价期 | 大促（如 Black Friday、Prime Day）前 2-4 周 | 平台需要备货，接受底线可临时下调 0.1-0.2 倍率 |
| 季度末冲量期 | 每季度最后 10-15 天 | 买手背 GMV KPI，低价通过率反而提升 |
| 淡季清库期 | 1-2 月、7-8 月 | 平台压价意愿强，非急需 SKU 可暂缓提报 |

插件节奏控制配置字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| rush\_period\_boost | 0.1 | 活动前锁价期，accept\_ratio 临时下调 0.1 |
| new\_cat\_bonus | true/false | 是否在新品窗口期启用更激进的首次申报倍率 |
| slow\_season\_pause | true/false | 淡季是否自动暂停非紧急 SKU 的核价流程 |
| retry\_interval\_hours | 24 | 同一 SKU 两次提报之间的最小间隔，避免被系统判定为刷屏 |

断供博弈的插件化记录：当运营手动下架某 SKU 后，插件记录 `suspension_date` ；重新上架时插件提示”该 SKU 曾有断供记录，建议重新核价”；重新核价时 accept\_ratio 可恢复至原配置，不继承断供前的失败倍率。

### 4.6.2 成本资料弹性与 BOM 拆分

当系统或买手进入”成本验证层”，要求提供成本证明时，插件需支持生成合规范围内的成本结构文档。

| 成本项 | 口径 | 备注 |
| --- | --- | --- |
| 原材料 | 按实际 BOM 拆分，加入 3%-5% 损耗系数 | 损耗系数需在合理范围内 |
| 加工费 | 分拆为注塑、组装、质检、包装等多道工序 | 工序拆分越细，总价越容易做高 |
| 包材费 | 内盒、外箱、标签、说明书、防潮袋逐项列示 | 包材升级（如彩盒替代白盒）可大幅增加 |
| 头程物流 | 按实际物流发票或预估价计入 | 含国内段、港杂、报关 |
| 质检预留 | 按 2%-5% 计入 | 对应退货和罚款预留 |
| 管理摊销 | 工厂管理费、设备折旧、人工社保等 | 适度摊入，不宜过高 |

执行口径：成本资料生成不是插件自动执行，而是插件在触发成本验证时调用预置模板生成；模板由运营团队维护，插件只负责填入当前 SKU 的 X、规格、重量等基础数据；生成的成本资料需人工审核后方可提交，插件不直接对外发送。

### 4.6.3 多主体矩阵与跨站点配置

同一款产品，在不同法人主体、不同店铺下向不同买手提报，可测试出核价上限。插件支持多主体配置，但不自动创建主体，只执行已配置主体间的策略调度。

多主体配置字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| entity\_pool | \[店铺 A, 店铺 B, 店铺 C\] | 同一款产品可投放的主体列表 |
| cross\_entity\_strategy | 顺序申报 / 并行申报 | 顺序：一家失败后换下一家；并行：同时向多家提报，取最优结果 |
| entity\_cooldown\_days | 7 | 同一主体失败后，冷却多少天才能再次提报同款 |

跨站点配置字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| site\_pool | \[US, EU, JP\] | 可投放的站点列表 |
| site\_priority | US > EU > JP | 站点优先级，优先在核价宽松的站点跑通 |
| price\_currency | USD / EUR / JPY | 各站点分别计价，插件按汇率换算后统一记录 |

执行边界：多主体/跨站点策略存在关联风控风险，插件需在日志中明确标记同一产品的所有提报主体和站点；若平台返回”关联店铺违规”，立即暂停该 `entity_pool` 下所有主体的同款提报，转人工。

### 五、异常处理与数据沉淀

### 5.1 人工复核触发条件

| 触发条件 | 处理方式 |
| --- | --- |
| 低价品 Pn < X + 6 | 停止复核，人工判断继续或放弃 |
| 高价品倍率阶梯走完仍未通过 | 人工判断继续或放弃 |
| R 低于接受底线 | 人工判断是否接受低价或调整策略 |
| SKU 采购价缺失 | 补充采购价后再核价 |
| 店铺 / 店铺组缺少 quote\_steps 配置 | 补配置后再核价 |
| 系统返回原因不明确 | 记录异常，汇总给产品或开发排查 |

### 5.2 日志与数据沉淀

**单 SKU 核价日志字段**

| 字段 | 说明 |
| --- | --- |
| sku\_id | SKU 唯一标识 |
| x | 采购价 |
| entity | 提报主体 |
| site | 提报站点 |
| round | 当前轮次 |
| strategy | 低价品 / 高价品 |
| p | 当前申报价 |
| r | 系统返回值 |
| r\_reason | 系统返回原因（如平台有提供） |
| action | 通过 / 继续复核 / 换图 / 改属性 / 转人工 / 放弃 |
| image\_version | 当前使用的图包版本 |
| sku\_version | 当前 SKU 属性版本 |
| timestamp | 提报时间戳 |

**数据沉淀用途**

- **图包效果分析** ：统计哪类主图角度/场景的通过率更高，反向优化图包制作。
- **买手画像** ：同一主体下不同类目的核价松紧度，辅助判断买手风格。
- **倍率阶梯优化** ：统计高价品各档倍率的实际通过率，动态调整 quote\_steps。
- **时间窗口验证** ：对比不同时段的核价结果，验证时间窗口策略的有效性。

**日志查询接**

插件需提供以下查询能力：

- 按 SKU 查询完整核价历史。
- 按店铺/店铺组查询当日/当周核价汇总。
- 按图包版本查询换图成功率。
- 按时间窗口查询通过率趋势。
